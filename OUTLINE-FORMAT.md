# 提纲格式

本文档定义提纲状态（`Outline State`）、证据处理、停止条件验证，以及 `perspective-outline-griller` 的状态持久化假设。

## 状态假设

最小可行版本（MVP）默认把当前对话上下文作为主要状态存储。不强制依赖外部数据库，也不假设存在隐藏的长期记忆。

为避免长对话中的状态漂移：

- 每经过 3 次用户回答，简要总结一次当前状态。
- 最终输出前，展示状态摘要，请用户纠正错误假设。
- 如果用户要求保存状态，把 Markdown 快照写入 `.outline-state/{slug}.md`。
- 如果没有写入文件，也必须能从可见对话摘要中恢复当前状态。

## 字段状态

每个字段只能使用一个状态：

| 状态 | 含义 |
| --- | --- |
| `missing` | 还没有可用值。 |
| `assumed` | 有工作假设，但尚未确认。 |
| `proposed` | 助手提出了推荐值，正在等待用户反馈。 |
| `confirmed` | 用户已确认，或证据已经实际核验。 |
| `conflicted` | 当前对话中存在互相冲突的值。 |

## 冲突解决规则

字段进入 `conflicted` 时，必须立即暂停其他字段推进。

触发条件：

- 用户给出与已确认字段不兼容的新说法。
- 两个已确认字段之间产生逻辑冲突。
- 视角切换导致原字段值不再适用于新受众或新目标。

处理步骤：

1. 把冲突字段标为 `conflicted`。
2. 明确展示冲突的两个版本。
3. 要求用户选择一个版本，或给出第三个新版本。
4. 用户确认后，把字段改为 `confirmed` 或 `assumed`。
5. 不得跳过 `conflicted` 字段直接输出提纲。

展示格式：

```md
这里有一个冲突需要先定：

- 版本 A：...
- 版本 B：...

我的建议：...

你要保留哪一个，还是给一个新版本？
```

## 字段记录

```md
- field:
  value:
  status: missing | assumed | proposed | confirmed | conflicted
  source: user | assistant_recommendation | evidence_handle | verified_source | sketch
  confidence: low | medium | high
  notes:
```

`source: sketch` 表示该字段来自临时草图视角推断，而不是正式人物 Skill 或用户确认。它的置信度默认不高于 `medium`，也不能直接等同于 `assistant_recommendation`，因为它还带有人物视角未正式蒸馏的质量限制。

## 核心字段

最终输出前必须清楚：

- Topic
- Perspective（人物视角）
- Audience
- Desired judgment
- Core claim
- Logic spine
- Sections
- Next artifact

## 扩展字段

按需启用：

- Evidence（证据）
- Scenarios（场景）
- Trade-offs（取舍）
- Risks（风险）
- Objections（反对意见）
- Terms to clarify（待澄清术语）
- Style constraints（风格约束）
- Stakeholders（相关方）
- Distribution channel（分发渠道）

## 视角字段

由适配器生成：

- Active Lens entries（启用中的镜片）
- Lens gaps（镜片缺口）
- Lens verification status（镜片验证状态）
- Perspective-specific stop criteria（视角专属停止条件）
- Adapter status（适配器状态）：`valid | stale | sketch`

## 证据规则

证据（`Evidence`）不表示助手“神奇地验证了事实”。它只表示提纲中有具体支撑句柄，或当前任务中确实完成了核验。

只有以下情况，证据才能标为 `confirmed`：

- 用户提供了具体证据句柄：文件名、数据表、报告名、内部系统行、引用、案例、截图或会议记录。
- 助手在当前任务中实际检查了本地文件、数据源或网页结果。
- 用户明确确认某个具体事实，并接受它作为素材来源。

以下情况只能标为 `assumed`：

- 用户说“我有内部数据”，但没有提供文件、指标或来源名。
- 用户说某个事实“应该是对的”或“后面可以找”。
- 助手提出了一个可能的例子，但没有核验。

以下情况标为 `missing`：

- 没有任何具体支撑句柄。
- 该判断依赖当前事实，但用户和助手都没有检查。

证据不能单独确认 `Core claim`。如果核心判断依赖证据，而证据是 `missing`，核心判断不得标为 `confirmed`。

## 通用停止条件

| 条件 | 验证问题 | 通过标准 | 判断主体 |
| --- | --- | --- | --- |
| 受众清楚 | 这份提纲写给谁？ | 能说出具体角色、场景和关心点。 | 用户 |
| 目标判断清楚 | 读完后要让对方相信或决定什么？ | 有一句话判断。 | 用户 |
| 逻辑主线清楚 | 所有章节是否服务同一条推理？ | 章节形成因果、递进、对比或论证，而不是并列堆点。 | 助手 + 用户 |
| 术语清楚 | 哪些词含混或多义？ | 关键术语有操作性定义，或被删除。 | 助手 + 用户 |
| 证据足够 | 每个关键判断靠什么支撑？ | 每个关键判断都有已确认证据、带标签的假设证据，或明确待补材料。 | 证据 + 用户 |
| 取舍清楚 | 这份提纲不讲什么？ | 边界和优先级明确。 | 用户 |
| 风险处理 | 聪明的反对者会质疑什么？ | 主要反对意见有回应，或被明确标为未解决。 | 助手 + 用户 |
| 下一步清楚 | 下一步要产出什么？ | 明确是文章、PPT、演讲稿、会议提纲或其他产物。 | 用户 |

## 停止规则

只有满足以下条件，才能输出最终版：

- 没有任何核心字段处于 `conflicted`。
- 所有核心字段都是 `confirmed`，或用户明确接受某些字段仍为 `assumed`。
- 高优先级视角镜片已经 `confirmed`，或用户明确接受为 `assumed`。
- 依赖证据的判断都有证据句柄、已核验来源，或可见的待补材料标签。

否则只能输出阶段稿。

## 阶段稿触发规则

用户要求输出时：

- 如果核心字段全部至少是 `assumed`，可以输出阶段稿，并显著标明工作假设。
- 如果任一核心字段仍是 `missing`，先追问最关键的 `missing` 字段，再询问用户是否仍要阶段稿。
- 如果任一核心字段是 `conflicted`，必须先解决冲突，不得输出阶段稿或最终稿。

## 阶段稿格式

```md
这是阶段版，不是最终共识版。

已确认：
- ...

工作假设：
- ...

证据状态：
- confirmed:
- assumed:
- missing:

明显缺口：
- ...

下一轮最该追问：
- ...
```

## 最终输出格式

```md
# 共识版提纲

## 核心判断

## 目标读者

## 逻辑主线

## 章节结构

## 关键证据与场景

## 取舍与边界

## 风险和反对意见

## 当前人物视角下的质量检查

## 仍需补充的材料

## 可直接进入写作的下一步提示
```

## 按产物适配输出

最终输出必须服务 `Next artifact`，不要只给通用目录。

如果 `Next artifact` 是 PPT，最终输出应加入页级结构：

```md
## 章节结构

### 第 1 页：{页标题}

- 要让读者接受的判断：
- 页面主文案：
- 口头讲解要点：
- 需要的素材或截图：
- 风险边界提醒：
```

如果 `Next artifact` 是 PRD，最终输出应加入产品结构：

```md
## 产品方案结构

- 目标用户：
- 核心问题：
- 使用场景：
- MVP 范围：
- 不做什么：
- 核心流程：
- 指标：
- 风险和待验证假设：
```

如果 `Next artifact` 是汇报、文章或会议提纲，按对应产物调整小节，但必须保留：

- 核心判断
- 目标读者
- 逻辑主线
- 关键证据与场景
- 取舍与边界
- 下一步提示
